# Hybrid Architecture Plan: Monolithic Core + Microservice Expansion

This proposed "hybrid" plan is an excellent middle ground. It prevents the current Intranet Portal from becoming an unmanageable monolith while avoiding the excessive overhead of retroactively splitting tightly-coupled existing modules (like Core & Assets) into completely separate repositories.

## Overview
Under this model, your deployment will look like this:

1. **The Legacy Monolith (Existing Codebase)**:
    - Contains **Core Module** (Auth, RBAC, Users, Dashboards, Global Search).
    - Contains **Assets Management Module**.
    - Reorganized internally into strict, domain-driven module folders (e.g., `frontend/src/app/(core)/` vs `frontend/src/app/(assets)/`).
    - Deployed to its existing frontend URL (`intranet.yourdomain.com`) and backend API URL (`api.intranet.yourdomain.com`).

2. **The Micro-Module Services (Future Codebases)**:
    - Example: *Drink Ordering Module*.
    - Completely separate codebase (Frontend Next.js app + Backend .NET API).
    - Deployed on its own subdomains (e.g., frontend: `drinks.intranet.yourdomain.com`, backend: `api-drinks.intranet.yourdomain.com`).
    - Uses JWT/OIDC provided by the Core Module to authenticate users.

## User Review Required

> [!WARNING]
> **Authentication & Authorization**
> Since new modules will live on separate URLs and have separate APIs, how will they handle user authentication?
> We will need the **Core Module** to act as an Identity Provider (like IdentityServer) or rely on a shared JWT secret so the Drink Ordering API can validate tokens generated by the Core API.

> [!IMPORTANT]
> **UI Consistency (The App Shell)**
> Moving to separate frontend URLs (e.g., `drinks.intranet.com`) means the user navigates away from the main portal. Should we:
> 1. Duplicate the Navigation Sidebar and Header in the new repository?
> 2. Expose the Drink Ordering frontend via a reverse proxy (e.g., Nginx maps `/drinks` to the separate service) so it feels like the same website?

## Proposed Changes (Phase 1: Existing Codebase Segregation)

Before we can build external modules, we must ensure the existing monolith's boundaries are strictly enforced.

### Frontend Module Segregation
We will reorganize the Next.js `src/app` router to use Route Groups to logically separate existing pages without changing the URL structure.

#### [NEW] `frontend/src/app/(core)/`
- Move `/login`, `/dashboard`, `/admin/users`, `/admin/roles` into this route group.
- Keep core components under `frontend/src/components/core/`.

#### [NEW] `frontend/src/app/(assets)/`
- Move `/assets`, `/admin/assets-management/` into this route group.
- Move asset-specific components to `frontend/src/components/assets/`.

### Backend Module Segregation
The .NET API and Data layers will be refactored to use a "Modular Monolith" folder structure.

#### [NEW] `backend/IntranetPortal.Api/Controllers/Core/`
- Contains `AuthController.cs`, `UsersController.cs`, `RolesController.cs`.

#### [NEW] `backend/IntranetPortal.Api/Controllers/Assets/`
- Contains `AssetsController.cs`, `CategoriesController.cs`.

#### [NEW] `backend/IntranetPortal.Data/Models/Core/` & `.../Models/Assets/`
- Segregate Models and Entity Framework configurations.

## Architectural Decisions

1. **Database Strategy**: Confirmed. New modules built outside the monolith will have their own completely **separate databases**. Modules kept inside the monolith (Core, Assets) will share the main database but have their codebase reorganized cleanly.
2. **Reverse Proxy vs Subdomains**: Deferred. We will decide on how to route the external modules' frontends at a later stage.

## Verification Plan

### Automated Tests
- Run `npm run build` after relocating Next.js route groups to ensure all paths still resolve.
- Run `dotnet build` after moving Controllers and Models.

### Manual Verification
- Start the server locally and verify that Assets Management operates perfectly when its files are completely isolated.
- Mock a JWT token from Core and verify that a standalone API can read it successfully.
