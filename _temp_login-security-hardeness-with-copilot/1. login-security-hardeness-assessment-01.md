# Login Workflow Security Assessment

Date: 2026-04-16  
Scope: Frontend login flow, backend auth API, and database-related auth model/configuration.

## Overall Verdict

The authentication flow has good foundations (JWT validation, BCrypt hashing, and server-side HttpOnly cookie handling), but it is not production-secure yet due to several high/critical issues. The most urgent gaps are hardcoded default credentials, lack of IsActive enforcement during login/authorization checks, and weak password policy enforcement.

## Findings (Ordered by Severity)

### 1) Critical: Known default credentials are present across UI and backend provisioning

- Login page pre-fills admin email/password:
	- frontend/src/app/(core)/login/page.tsx:52
	- frontend/src/app/(core)/login/page.tsx:69
- Startup seeding always hashes a hardcoded admin password:
	- backend/IntranetPortal.Api/Program.cs:115
- Seeder creates/maintains `admin@company.com` using the seeded hash:
	- backend/IntranetPortal.Data/Data/DatabaseSeeder.cs:44
	- backend/IntranetPortal.Data/Data/DatabaseSeeder.cs:48
	- backend/IntranetPortal.Data/Data/DatabaseSeeder.cs:49
- Additional hardcoded default passwords for newly created login accounts:
	- backend/IntranetPortal.Api/Controllers/Core/EmployeesController.cs:118
	- backend/IntranetPortal.Api/Controllers/Core/EmployeesController.cs:161
	- backend/IntranetPortal.Api/Controllers/Core/SetupController.cs:298
	- backend/IntranetPortal.Api/Controllers/Core/SetupController.cs:316

Security impact:
- First-login compromise risk is high if these defaults are known or reused in non-local environments.

### 2) High: Disabled accounts can still authenticate and keep access until token expiry

- Login path does not enforce `IsActive` before issuing JWT:
	- backend/IntranetPortal.Api/Controllers/Core/AuthController.cs:37
	- backend/IntranetPortal.Api/Controllers/Core/AuthController.cs:39

	- backend/IntranetPortal.Api/Controllers/Core/AuthController.cs:202

### 3) High: Weak password policy at reset endpoint

- Password reset only checks for length >= 6:
	- backend/IntranetPortal.Api/Controllers/Core/UsersController.cs:139

	- backend/IntranetPortal.Api/appsettings.Development.json:15

## Recommended Changes (Prioritized)

1. Remove all hardcoded credentials/default passwords immediately.
- Remove UI prefilled credentials.
- Replace static seeded admin password with environment-provided bootstrap secret or one-time setup flow.
- Remove `Welcome2026!` from create/import/setup flows.

2. Enforce account activity (`IsActive`) consistently.
- Add `IsActive` checks in login query and permission checks.
- Consider token versioning/revocation to invalidate existing sessions on disable or reset.

3. Upgrade password policy and centralize validation.
- Minimum 12 chars.
- Require mixed character classes.
- Reject common/breached patterns.
- Reuse one shared validator for reset/create/import paths.

4. Move secrets out of tracked config and rotate exposed values.
- Use environment variables/secret manager for JWT keys and DB credentials.
- Rotate any leaked development or shared secrets.

5. Add DB-level constraints for account identity.
- Add normalized email column or write-time normalization.
- Add unique index on normalized email in `Core_UserAccounts`.

6. Tighten cookie domain strategy.
- Default to host-only cookie.
- Enable wildcard domain only for explicitly trusted deployment patterns.

7. Add auth abuse controls.
- Rate limiting on login endpoint.
- Lockout or exponential backoff on repeated failures.
- Structured audit logs for login attempts and password resets.

## Open Questions

3. Which secret management approach should be standard (env vars only vs vault)?

## Suggested Implementation Sequence

Phase 1 (Immediate hardening)
- Remove hardcoded defaults and prefilled UI credentials.
- Enforce `IsActive` checks in auth and authorization paths.
- Strengthen reset password policy.

Phase 2 (Data and configuration hardening)
- Move secrets to environment/vault and rotate exposed keys/passwords.
- Add unique normalized email constraint + migration.

Phase 3 (Operational protection)
- Add login rate limiting/lockout.
- Add session revocation strategy and security audit trail coverage.
