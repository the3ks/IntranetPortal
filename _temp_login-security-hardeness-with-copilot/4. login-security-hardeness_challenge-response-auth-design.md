# Challenge-Response Password Transfer Design

Date: 2026-04-16
Scope: IntranetPortal monolith auth flow

Implementation Status: Phase 1 implemented on 2026-04-16

## Security Position

- This flow reduces direct password exposure in transit payloads.
- This flow does NOT replace TLS.
- TLS remains mandatory because active MITM can still proxy challenge-response exchanges.

## Protocol Direction

- Implemented now: one-time challenge + RSA-OAEP encrypted password transfer.
- Future upgrade target: OPAQUE-style PAKE (or SRP variant if OPAQUE libs are unavailable).
- Do not implement static client-side hashing as a standalone protection.

## High-Level Flow

1. Client requests challenge with normalized email.
2. Server returns one-time challenge payload:
   - nonce
   - challengeId
   - issuedAt
   - expiresAt (30-60s)
   - algorithm metadata
3. Client encrypts password with server public key (Web Crypto RSA-OAEP-256).
4. Client submits challengeId + encryptedPassword.
5. Server decrypts payload, verifies password via BCrypt, and checks one-time/expiry.
6. On success, server issues JWT exactly as current flow (with SecurityStamp claim).

## Required Server Controls

- One-time challenge storage and consume-on-success semantics.
- Replay protection keyed by challengeId + consumed flag + expiry.
- Strict expiration (30-60s).
- Rate limiting on both issue and verify endpoints.
- Preserve existing lockout and inactive-account checks.
- Audit event logging for challenge issue/verify failures.

## API Surface (Core Auth)

- POST /api/auth/challenge/start
  - Input DTO: email
  - Output DTO: challengeId, nonce, expiresAt, algorithm, publicKeyPem

- POST /api/auth/challenge/complete
  - Input DTO: challengeId, email, encryptedPassword
  - Output DTO: token, role

## Data Model Additions

- New table: Core_LoginChallenges (implemented)
  - Id
  - ChallengeId (unique)
  - UserAccountId
  - Nonce
  - IssuedAt
  - ExpiresAt
  - ConsumedAt (nullable)
  - ClientIpHash (optional)
  - UserAgentHash (optional)

## Frontend Notes

- Keep current login path as fallback while rolling out challenge flow.
- Feature flag: ENABLE_CHALLENGE_LOGIN=true enables challenge flow in server action.
- Use Web Crypto APIs for client operations.
- Avoid storing reusable password-derived material in local storage.

## Rollout Plan

1. Add challenge endpoints behind feature flag. (Done)
2. Test with internal users only.
3. Monitor failed verifications, lockouts, and latency.
4. Gradually migrate login UI to challenge flow.
5. Keep TLS enforced at gateway and app levels.

## Non-Goals

- No support for plaintext HTTP.
- No claim that challenge-response alone secures authentication without TLS.
